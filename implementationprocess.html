X Connect registration and validation of the terms and condition of use

The service provider must register to X Connect via the registration form made available at http://xonnect.e-monsite.com/contact/sign-up.html. As part of the implementation, the service provider must register specifying the following information:

Name of the service Mandatory
Contact Email Mandatory
Logo of service Mandatory
Callback Urls - one per line Mandatory
Your use case Mandatory

* Additional information on callback URLs:

  • The URL can be any depth subdomains
  • The service provider can use specific ports
  • The service provider can use an IP rather than FQDN
  • The "search" of the URL may be an arbitrary size
  • The service provider can use "localhost" for its development environments and test:

To validate his registration the service provider must accept the terms and conditions of use of X Connect service when sending its information.

Integration and configuration of an OpenID Connect client

X Connect follows the standard OpenID Connect implementation. The OpenID Connect protocol is an identification overlay on top of the OAuth 2.0 protocol. It allows a service provider to access the pivotal identity of a user, transmitted by an authentication provider through X Connect.

* Additional information about IDOC:

  • Protocol specification: http://openid.net/connect/.
  • Reference OpenID Connect implementation: http://openid.net/developers/libraries/

The service provider is an OpenID Connect relying party to X Connect

X Connect provider is an OpenID Connect identity provider for the service provider

Integration

The service provider must implement and configure an OpenID Client client to communicate with X Connect (considered OIDC provider).

Configuration

The OpenID Connect protocol defines three REST calls and one endpoint by OIDC client (three endpoints on provider side)
The endpoints are available in integration environment here:
Authorization https://x-connect.herokuapp.com/authorize
Token https://x-connect.herokuapp.com/token
UserInfo https://x-connect.herokuapp.com/userinfo
Logout https://x-connect.herokuapp.com/logout

Kinematic

The service provider in its implementation must follow the kinematics:

            1. The user clicks on the client authentication button. 

            2. The client redirects to the "authorization endpoint" of the provider with a customer id and callback url.

Implementing the disconnection

Account reconciliation and dissociation

Fiatope Digital Solutions recommends implementing reconciliation and dissociation of existing accounts of Service Provider users. The implementation is specific to the Service Provider according to its Information System.

Error Handling between XConnect and Service Provider

As an OpenID Connect Provider, XConnect can send all kinds of errors to a client application . To do so, XConnect goes through the feedback mechanism errors of an OpenID Connect identity provider as described in the standard (http://openid.net/specs/openid-connect-core- 1_0.html # authError, particularly the sections 3.1.2.6 (authentication) 3.1.3.4 (access token), 5.3.3 (User information service))

Expiration of Data

XConnect manages several types of "perishable" data at the time of the OpenID Connect authentication, or at the time of providing an access token to a protected resource (kinematic OAuth2 classic). Each of these data has a life of its own beyond which it is be regenerated

Type Usage Time
Web session At every authentication and to maintain the session at XConnect side 30 minutes without action
Access Token Retrieval of information (phase 3 authenticiation kinematic/COAuth2 kinematic) 20 minutes
Authorization code Code provided at the beginning of the authentication process, it is then used to retrieve the access token 5 minutes
